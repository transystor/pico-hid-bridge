# Pico SDK migration plan

План перехода с CircuitPython MVP на более надёжную реализацию через Pico SDK.

## Почему уходим с CircuitPython

На практике в нашем сценарии проявилось нестабильное поведение Wi‑Fi:
- `wifi.radio.connect(...)` кидает `ConnectionError('Неизвестный сбой 1')`,
- DHCP lease на роутере иногда появляется,
- но интерфейс на устройстве не удерживается,
- TCP listener до рабочего состояния не доходит.

То есть проблема уже не в нашей бизнес-логике, а в хрупкости runtime для данного сценария.

## Целевой стек

### Pico W
- Pico SDK
- C/C++
- TinyUSB device stack
- cyw43_arch
- lwIP

### Raspberry Pi 4
- orchestration/API layer
- TCP client to Pico W
- позже можно делать на .NET

## Цели по этапам

### Этап 1
Собрать минимальный проект Pico SDK:
- CMake
- main.c
- TinyUSB descriptors
- базовый цикл `tud_task()`

### Этап 2
Подтвердить, что Pico W определяется как HID mouse на ПК.

### Этап 3
Поднять Wi‑Fi STA mode и стабильное подключение.

### Этап 4
Поднять lwIP TCP listener на порту `5000`.

### Этап 5
Принять первую текстовую команду:
- `PING`
- `MOVE 10 0`

### Этап 6
Добавить очередь HID-событий и реальную отправку mouse report.

## Важное архитектурное решение

Pico W должен оставаться максимально простым исполнителем:
- не хранить тяжёлое состояние сессии,
- не принимать архитектурные решения,
- не держать сложный API.

Вся orchestration-логика должна жить на Raspberry Pi 4 или выше.
